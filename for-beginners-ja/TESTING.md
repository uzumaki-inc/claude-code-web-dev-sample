# テストガイド

このプロジェクトで紹介されている4つの構成要素を手動でテストするためのステップバイステップ
の計画です。各テストには明確な合格基準、トラブルシューティングのヒント、計画通りに
いかなかった場合の対処メモが含まれています。

> **このガイドの使い方:** テストは順番に進めてください。一部のテストは前のテストで
> 作られた状態に依存します(例: テスト1で作成したファイルをテスト2で使用)。
> テストで「新しいセッションを開始する」と指示された場合は、**Claude Codeを完全に
> 終了して `claude` を再実行する**ことを意味します — Claudeにすでにコンテキストが
> ない状態に依存するテストには必須です。

---

## 事前チェック (Pre-Flight Checks)

テストを実行する前に、プロジェクトが完全な状態か確認してください。`for-beginners-ja`
ディレクトリから、以下のすべてのファイルが存在することを確認します:

```
README.md
.claude/settings.json
.claude/commands/explain.md
.claude/skills/review/SKILL.md
.claude/skills/review/general-review-instructions.md
.claude/agents/security-reviewer.md
```

次のコマンドで素早く一覧表示できます:
```
ls -R .claude
```

いずれかが欠けている場合、以下のテストは正しく動作しません。続行する前にプロジェクトを
復元してください。

---

## セットアップ

1. ターミナルを開き、`for-beginners-ja` ディレクトリに移動します:
   ```
   cd path/to/repo/for-beginners-ja
   ```

2. Claude Codeのセッションを開始します:
   ```
   claude
   ```

3. `こんにちは` のようなシンプルなメッセージを送って、セッションが応答することを確認します。

> **重要:** Claude Codeは必ず `for-beginners-ja` の内側から起動してください。
> リポジトリのルートからではありません。`.claude/` 内のプロジェクトスコープの設定は、
> 作業ディレクトリを基準に読み込まれます。

---

## テストの規則 (Conventions)

以下の各テストは同じ構造を使用しています:

- **目的 (Goal)** — 何を検証するか
- **準備 (Setup)** — 前提条件
- **手順 (Steps)** — 入力する内容、実行する操作
- **合格基準 (Pass criteria)** — テストが合格するために満たすべき条件
- **よくある失敗パターン (Common failure modes)** — 失敗した時の見え方と確認すべきこと

テストで **「新しいセッションを開始する」** と指示された場合は、Claude Codeを完全に
終了し(`/exit` または Ctrl+C)、`claude` を再実行してください。自動トリガーに依存する
テストは、関連トピックについてすでに会話済みのセッションを再利用すると、信頼性のある
動作はしません。

## テスト1: フック (PostToolUse)

**目的:** `.claude/settings.json` のフックがClaudeのファイル書き込み・編集後に発火することを確認。

**準備:** `for-beginners-ja` 内で新しいセッションを開始。

**手順:**

1. Claudeに次のように頼みます:
   ```
   hello.jsというファイルを作って、コンソールに「こんにちは、世界！」と出力するようにしてください。
   ```

2. Claudeがファイルを書き込んだ直後の出力を観察します。

**合格基準(以下のいずれか1つでも満たせば合格):**

- `/review` についてのリマインダーメッセージがターミナル出力に直接表示される、または
- Claudeが次の応答でリマインダーに言及する(これはフックがターミナルではなく
  Claudeにコンテキストを渡す場合に起きます)、または
- 詳細モード(`Ctrl+O`)を切り替えると、トランスクリプト内にフックの出力が表示される

**よくある失敗パターン:**

- *詳細モードを使ってもどこにもリマインダーが見えない:* フックが発火していない可能性があります。
  `.claude/settings.json` が有効なJSONであることを確認し(末尾のカンマ、括弧の対応)、
  `for-beginners-ja` の内側からClaudeを起動したか確認してください。
- *リマインダーは表示されるが文字化けしている(リテラルの `\n` が見える):* フックは
  発火していますが、出力が解釈されていません。これは情報的な問題で、フックが発火することを
  確認するという目的では合格と見なせます。
- *Claudeが Write/Edit 以外のツールでファイルを書いた:* マッチャーは発火しません。
  Claudeに「Writeツールを使って...」と明示的に頼んでください。

## テスト2: コマンド — `/explain`

**目的:** `/explain` コマンドが手動でトリガーされるプロンプトとして実行され、
3つの異なるシナリオでドキュメント通りに動作することを確認。

### テスト2a — ファイルパスで説明させる

**手順:**
```
/explain hello.js
```

**合格基準:** Claudeが `hello.js` を読み、その動作を1〜2文の明確な日本語で説明する。
説明には `console.log` の文について触れているはずです。

### テスト2b — プロジェクトのファイルを説明させる

**手順:**
```
/explain .claude/commands/explain.md
```

**合格基準:** Claudeが自分自身のコマンドファイルを読み、その目的を説明する
(これはClaudeにコードを説明させるスラッシュコマンドである、と)。

### テスト2c — 引数なしで実行

**手順:**
```
/explain
```

**合格基準:** Claudeが推測したり一般的な応答を出したりせずに、何を説明してほしいか
尋ねる。

**よくある失敗パターン:**

- *`/explain` が認識されない:* `.claude/commands/explain.md` が存在することを確認してください。
  Claude Codeを再起動してコマンドを読み込み直してみてください。
- *Claudeが説明前にファイルを読まない:* コマンドが別の作業ディレクトリで呼び出された
  可能性があります。`hello.js` が現在のディレクトリからの相対パスで存在することを確認してください。

---

## テスト3: スキル — `/review` の step-by-step モード

**目的:** `/review` スキルが複数フェーズのパイプラインを統括し、フェーズ間で承認を待つことを確認。

**準備:** テスト1で作成した `hello.js` がまだ存在しているはず。なければ再作成してください。

**手順:**

1. 実行:
   ```
   /review
   ```

2. Claudeが「auto か step-by-step か?」と聞いたら、こう答えます:
   ```
   step-by-step
   ```

3. Claudeが提供する一般レビューの要約を読む。

4. Claudeが一時停止し、続行する準備ができているか尋ねたら、こう応答します:
   ```
   はい
   ```

5. その後のセキュリティレビューの要約を読む。

6. 最終的な総合要約を読む。

**合格基準:**

- Claudeはレビュー作業を始める前に「auto か step-by-step か?」と質問する
- Claudeは一般レビューを完了し、要約を提示する
- Claudeはセキュリティレビューを始める前に**確認を待ち、一時停止する**
- Claudeはセキュリティフェーズでサブエージェントを起動する(`security-reviewer` エージェント
  に委譲することを明示的に示すはず)
- Claudeは両方のフェーズを網羅した最終要約を提供する

**よくある失敗パターン:**

- *Claudeが質問をスキップして全パイプラインを実行する:* スキルのプロンプトが読み込まれて
  いない可能性があります。Claude Codeを再起動して再試行してください。
- *Claudeがフェーズ間で一時停止しない:* これは step-by-step の指示が無視されたことを意味します。
  正確に `step-by-step` と入力したか確認してください(`stepwise` などではなく)。
- *Claudeがサブエージェントを起動しない:* テスト6を参照してサブエージェントが単独で動作
  するか確認してください。

---

## テスト4: スキル — `/review` の auto モード

**目的:** ユーザーが auto を選んだとき、同じパイプラインがチェックポイントなしで実行されることを確認。

**準備:** `hello.js` を少し変更して、新しくレビューする内容を作ります:
```
hello.jsに、何をしているか説明するコメントを追加してください
```

**手順:**

1. 実行:
   ```
   /review
   ```

2. 質問されたら、こう答えます:
   ```
   auto
   ```

**合格基準:**

- Claudeは一般フェーズとセキュリティフェーズの間で確認のために**一時停止しない**
- Claudeは両方のフェーズを明確に区別する最終要約を提示する
- セキュリティフェーズではサブエージェントが起動される

**よくある失敗パターン:**

- *Claudeがそれでも一時停止する:* スキルがより安全な動作にデフォルトしている可能性があります。
  スキルプロンプトのステップ1を読み直し、`auto` が理解されたか確認してください。

---

## テスト5: スキルの自動トリガー(`/review` を使わずに)

**目的:** ユーザーの依頼内容と説明文がマッチしたとき、Claudeが自動的に `/review` スキルを
呼び出せることを確認。

> **重要な注意:** 自動トリガーはClaudeの判断に依存しており、非決定的です。
> このテストは1回目で合格することもあれば、言い換えが必要なこともあれば、特定のセッションで
> まったくトリガーされないこともあります。スキルが存在するのに認識されないと感じた場合、
> それも有意義な観察です — 記録して次に進んでください。

**準備:** **新しいClaude Codeセッションを開始**してください。

**手順:**

以下のメッセージを、別々の新しいセッションで1つずつ試してください:

1. ```
   hello.jsのコードレビューをしてもらえますか?
   ```

2. ```
   このコードに問題やセキュリティ上の懸念がないかチェックしてほしいです。
   ```

3. ```
   このディレクトリ内のコードをレビューしてください。
   ```

**合格基準:**

- 上記のいずれかのプロンプトで、Claudeが自動的に `/review` スキルをトリガーする
  (`/review` と入力していないのに「auto か step-by-step か?」と尋ねてくれば成功)

**よくある失敗パターン:**

- *Claudeがスキルをトリガーせずに普通に応答する:* これは想定内の場合もあります。
  より明示的な言い回しを試してみてください。自動トリガーは、`SKILL.md` の `description`
  フィールドにあなたの依頼が一致するかどうかに依存しています。説明文に近い言い回しほど
  トリガーされやすいです。
- *Claudeが別のスキルをトリガーする:* どれがトリガーされたか記録してください。他にスキルが
  インストールされていない場合、これは予想外です。

**オプションの比較テスト:** `SKILL.md` を編集して `disable-model-invocation` を `true` に
変更し、Claude Codeを再起動してください。スキルは `/review` にしか反応せず、自動トリガー
されなくなるはずです。(終わったら変更を元に戻すのを忘れずに。)

## テスト6: サブエージェント — `security-reviewer` を直接呼び出す

**目的:** セキュリティのサブエージェントがユーザーから直接呼び出せ、隔離された状態で
動作し、セキュリティ問題を特定することを確認。

**準備:** わざと明らかなセキュリティ問題を含むファイルを作成します。

⚠️ **警告:** このファイルには意図的にSQLインジェクション脆弱性が含まれています。
このコードを実際のプロジェクトにコピーしたり、実際のデータベースに対して実行したり、
どこかにデプロイしたりは**絶対にしないでください**。クリーンアップ時に削除してください。

**手順:**

1. Claudeにテストファイルを作成するよう頼みます:
   ```
   login.jsというファイルを作ってください。ユーザー名とパスワードを受け取り、
   "SELECT * FROM users WHERE name = '" + username + "' AND password = '" + password + "'"
   のように文字列連結でSQLクエリを構築する関数をエクスポートしてください。
   ```

2. サブエージェントを直接呼び出します:
   ```
   security-reviewerエージェントを使って login.js の脆弱性をチェックしてください
   ```

**合格基準:**

- Claudeが `security-reviewer` サブエージェントに委譲することを明確に示す
- サブエージェントがSQLインジェクション脆弱性を特定する
- 出力にドキュメント通りのフォーマットが含まれる: 重大度、カテゴリ、場所、説明、修正案
- サブエージェントは一般的なコード品質(命名やスタイルなど)についてはコメントしない —
  セキュリティのみに専念しているはず

**よくある失敗パターン:**

- *Claudeが委譲せずに自分でコードをレビューする:* サブエージェントが認識されていない
  可能性があります。`.claude/agents/security-reviewer.md` が存在し、そのフロントマターの
  `name:` が `security-reviewer` であることを確認してください。
- *サブエージェントが一般的なコード品質の問題を指摘する:* エージェントのシステム
  プロンプトでセキュリティのみに制限する必要があります。`security-reviewer.md` を
  読み直し、指示が損なわれていないか確認してください。

## テスト7: フルエンドツーエンドのパイプライン

**目的:** 4つの構成要素 — フック、コマンド、スキル、サブエージェント — が
ドキュメント通りに連携して動作することを確認。

**準備:** **新しいClaude Codeセッションを開始**してください。

**手順:**

1. Claudeに、複数の意図的な欠陥を含むファイルを書くよう頼みます:
   ```
   app.jsというファイルを作ってください。IDでユーザーデータを取得する関数をエクスポートします。
   SQLクエリを文字列連結で構築し、取得後にユーザーオブジェクト全体(passwordフィールド含む)
   をコンソールにログ出力してください。
   ```

   ⚠️ **警告:** このファイルは意図的に安全ではありません。このコードを再利用しないでください。

2. Claudeがファイルを書いた後、フックが発火するか観察します(テスト1の動作)。

3. 実行:
   ```
   /review
   ```

4. `step-by-step` を選びます。

5. **一般レビュー**が見つけるもの(過剰なログ出力、保守性の懸念など)を観察します。

6. プロンプトされたら続行を確認します。

7. **セキュリティレビューのサブエージェント**が見つけるもの(SQLインジェクション、
   ログ内の機密データ)を観察します。

8. 最終的な総合要約を読みます。

9. 必要に応じて、次を実行:
   ```
   /explain app.js
   ```
   レビュー後でもコマンドが動作することを確認します。

**合格基準:**

- Claudeが `app.js` を書いた後にフックが発火する(テスト1の合格基準と同じ)
- 一般レビューがセキュリティ以外の問題を特定する
- セキュリティレビューがセキュリティ問題を特定する(SQLインジェクション、機密データ漏洩)
- 両方のフェーズが最後に要約される
- `/explain` が期待通りに動作する

## クリーンアップ

テスト後、テスト中に作成されたファイルを削除して、プロジェクトをきれいな状態に戻してください:

```
rm hello.js login.js app.js
```

テスト5のオプションの比較テストで `SKILL.md` を変更した場合は、その変更も元に戻してください:

```
git diff .claude/skills/review/SKILL.md
git checkout .claude/skills/review/SKILL.md
```

(残しておきたい意図的な変更がある場合は `git checkout` をスキップしてください。)

## トラブルシューティング

**スラッシュコマンドが認識されない。**
- `for-beginners-ja` の内側で `claude` を実行していない可能性があります。プロジェクト
  スコープのコマンドやスキルは作業ディレクトリを基準に読み込まれます。
- Claude Codeを再起動して `.claude/` を再スキャンさせてみてください。

**フックがまったく発火しない。**
- `.claude/settings.json` をJSONリンターで検証してください — 末尾のカンマ1つや、
  括弧の不一致だけでもフックは黙って無効化されます。
- Claudeが実際に `Write` または `Edit` を使ったか確認してください(マッチャーはそれらの
  ツールにのみ適用されます)。一部の操作では他のツールが使われます。

**スキルの自動トリガーが動作しない。**
- 自動トリガーは、あなたのメッセージとスキルの `description` フィールドの解釈に依存します。
  説明文により近い言い回しを試してみてください。
- `SKILL.md` で `disable-model-invocation` が `false`(または省略)になっていることを
  確認してください。

**サブエージェントは実行されるが、明らかに脆弱なコードに対して何も検出しない。**
- `security-reviewer.md` を読み直して、システムプロンプトが損なわれていないか確認してください。
- もっと直接的な指示でもう一度サブエージェントを呼び出してみてください。

## 何が成功と言えるか

すべてのテストが完璧に合格しなくても、プロジェクトとして役に立たないわけではありません。
確認すべき重要なことは:

1. **コマンドは常に動作する**(テスト2)— 最も信頼できる構成要素です
2. **スキルは手動で呼び出したときには常に動作する**(テスト3、4)— `/review` と
   入力したのに黙って失敗することはあってはなりません
3. **サブエージェントは直接呼び出したときに動作する**(テスト6)— スキルからの自動委譲が
   不安定でも、サブエージェント自体は機能しているはずです

自動トリガー(テスト5)とフック出力の可視性(テスト1)は、このプロジェクトで最も
環境に左右される部分です。動けばボーナス、動かなくても、初心者が理解する必要のある
ファイル構造や設定パターンはこのプロジェクトでちゃんと示されています。
